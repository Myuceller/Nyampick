# 냠픽 면접 준비 노트

목적: 냠픽 프로젝트를 면접에서 설명할 때 쓸 핵심 스토리, 기술 선택 이유, 트러블슈팅 답변을 정리한다.

취업 목표 기반 개선 방향은 [냠픽 취업 목표 기반 개선 로드맵](career-roadmap.md)을 기준으로 관리한다.

## 1분 프로젝트 소개

냠픽은 아이 식단 기록, 냉장고 재료 관리, 영수증 스캔, AI 레시피 추천을 연결한 모바일 중심 식단 관리 웹앱입니다. 보호자가 매일 아이가 먹은 음식과 반응을 기록하고, 냉장고에 있는 재료를 기준으로 유아식 메뉴를 추천받을 수 있게 만들었습니다.

기술적으로는 Next.js App Router, TypeScript, Supabase Auth/Database, OpenAI API, Vercel 배포를 사용했습니다. 저는 로그인 기반 앱 흐름, 식단/냉장고/레시피 데이터 흐름, AI 추천 품질 검증, 배포 성능 개선, SEO 공개 페이지 정리를 중심으로 구현했습니다.

특히 배포 환경에서 API 응답이 로컬보다 크게 느린 문제를 측정했고, Vercel Function region을 한국과 가까운 `hnd1`로 고정해 주요 API 응답 시간을 줄였습니다. 이후 검색 노출을 위해 루트 `/`를 공개 랜딩으로 정리하고, 로그인 뒤 앱 홈은 `/meal`로 분리했습니다.

## 핵심 기능

| 기능 | 설명 | 면접에서 강조할 점 |
| --- | --- | --- |
| 식단 기록 | 아침, 점심, 저녁, 간식 기록과 주간 식단 확인 | 반복 사용 화면이라 UX와 데이터 저장 흐름이 중요했다. |
| 냉장고 관리 | 일반 재료, 이유식 큐브, 소스, 간식 분류 | 재료 상태가 AI 추천 입력으로 이어진다. |
| 영수증 스캔 | 사진 기반 재료 후보 추출 | 사용자의 입력 비용을 줄이는 기능이다. |
| AI 레시피 추천 | 냉장고 재료와 아이 정보를 바탕으로 추천 | AI 결과를 그대로 쓰지 않고 품질 기준을 둔다. |
| 가족/아기 관리 | 아이 정보, 가족 연동, 알레르기 정보 | 개인화와 권한 관리가 필요한 영역이다. |
| 공개 SEO 페이지 | 랜딩, 서비스 소개, 가이드 페이지 | 로그인 앱과 검색 노출 페이지를 분리했다. |

## 기술 스택 설명

### Next.js App Router

짧은 답변:

> React 기반으로 앱 화면을 만들면서도, 공개 페이지는 정적 HTML로 생성할 수 있어서 Next.js를 선택했습니다. 냠픽은 로그인 뒤 앱 화면과 검색 노출이 필요한 공개 페이지가 함께 있어서 App Router 구조가 잘 맞았습니다.

깊게 물었을 때:

- 공개 페이지는 SSG로 빌드 시점에 HTML을 만든다.
- 로그인 뒤 화면은 인증 상태와 클라이언트 상호작용이 중요해서 CSR 성격이 강하다.
- `metadata`, `sitemap.ts`, `robots.ts`를 코드로 관리해 SEO 설정을 일관되게 유지했다.
- `/`는 공개 랜딩, `/meal`은 로그인 뒤 식단 홈으로 분리했다.

냠픽 사례:

| 구분 | 라우트 |
| --- | --- |
| 공개 SEO 페이지 | `/`, `/about`, `/guide/*`, `/privacy`, `/terms` |
| 로그인 필요 앱 화면 | `/meal`, `/fridge`, `/recipe`, `/mypage`, `/children`, `/family` |
| 검색 제외 | `/api/*`, `/auth`, 로그인 뒤 기능 라우트 |

### SSR, CSR, SSG 구분

| 방식 | 설명 | 냠픽 예시 |
| --- | --- | --- |
| CSR | 브라우저에서 JS 실행 후 화면을 만든다. | `/meal`, `/fridge`, `/recipe`, `/mypage` |
| SSR | 요청마다 서버가 HTML을 만든다. | 실시간 공개 데이터 페이지가 있다면 후보 |
| SSG | 빌드할 때 HTML을 미리 만든다. | `/`, `/about`, `/guide/*` |

면접 답변:

> CSR은 브라우저에서 화면을 만드는 방식이라 로그인 후 상호작용 많은 앱 화면에 적합합니다. SSR은 요청 시점에 서버가 HTML을 만들어 최신 데이터를 보여줄 때 유리합니다. SSG는 빌드 시점에 HTML을 미리 만들어 빠르고 SEO에 유리합니다. 냠픽에서는 검색 노출이 필요한 랜딩과 가이드 페이지를 SSG 성격의 공개 페이지로 분리했고, 식단/냉장고 같은 인증 기반 기능은 CSR 중심으로 구성했습니다.

### Supabase

짧은 답변:

> 인증과 데이터베이스를 빠르게 연결할 수 있고, 사용자별 데이터 접근 모델을 만들기 쉬워 Supabase를 사용했습니다.

깊게 물었을 때:

- Supabase Auth로 로그인 세션을 관리한다.
- API Route에서 인증된 사용자 기준으로 식단, 냉장고, 가족 데이터를 조회한다.
- 서버 API에서는 service role key가 필요한 작업과 사용자 세션 기반 작업을 분리해야 한다.
- 민감한 key는 클라이언트에 노출하지 않는다.

주의해서 말할 점:

- Supabase를 썼다고 보안이 자동으로 끝나는 것은 아니다.
- API에서 인증 사용자 확인, family/child ownership 확인이 필요하다.
- 환경 변수와 service role key 노출 방지가 중요하다.

### OpenAI API

짧은 답변:

> 냉장고 재료와 아이 식단 맥락을 입력으로 받아 유아식 레시피 아이디어를 제안하는 데 사용했습니다. 단순 호출이 아니라 품질 평가 케이스를 두고 결과를 점검했습니다.

깊게 물었을 때:

- AI 결과는 확정 답이 아니라 보호자가 확인할 추천안으로 다룬다.
- 알레르기, 월령, 조리 안전 기준을 프롬프트와 결과 검증에서 고려한다.
- `tests/recipe-ai-quality-gate.test.ts`, `scripts/ai-recipe-quality-report.mjs` 같은 품질 점검 흐름을 둔다.
- 결과 품질은 정량 점수와 케이스별 리포트로 추적한다.

## SEO 개선 스토리

문제:

- 처음에는 `/`가 로그인 뒤 앱 홈으로 쓰였고, 비로그인 사용자는 앱 로딩/인증 흐름을 보게 됐다.
- 검색엔진에게 보여줄 대표 공개 페이지가 명확하지 않았다.
- 공개 가이드 페이지가 적어 일반 검색어 유입 가능성이 낮았다.

해결:

- `/`를 공개 랜딩으로 변경했다.
- 기존 식단 홈을 `/meal`로 이동했다.
- `sitemap.ts`에 공개 페이지를 등록했다.
- `robots.ts`에서 로그인 뒤 기능 라우트와 API를 크롤링 제외했다.
- `NEXT_PUBLIC_CANONICAL_URL=https://www.nyampick.kr` 기준으로 canonical/sitemap 도메인을 고정했다.
- 가이드 페이지를 추가했다.

추가한 공개 가이드:

- `/guide/baby-meal-plan`
- `/guide/baby-food-cube-storage`
- `/guide/fridge-ingredient-management`

면접 답변:

> Next.js를 쓴다고 자동으로 SEO가 좋아지는 것은 아니라고 생각합니다. 검색엔진이 읽을 수 있는 공개 HTML, canonical, sitemap, robots 정책이 같이 맞아야 합니다. 냠픽에서는 로그인 뒤 앱 화면과 공개 콘텐츠를 분리하고, 루트 `/`를 랜딩 페이지로 변경했습니다. 이후 sitemap에는 공개 페이지들만 넣고, `/meal`, `/fridge`, `/recipe` 같은 인증 기반 화면은 robots에서 제외했습니다.

## 성능 개선 스토리

문제:

- 로컬에서는 빠른데 배포 사이트에서는 탭 클릭 후 2-3초 기다리는 느낌이 있었다.
- Lighthouse 점수만 보면 나쁘지 않았지만, 실제 체감은 API 응답 시간이 문제였다.

측정:

- 로그인 세션으로 보호 라우트와 API 응답 시간을 측정했다.
- 배포 API가 로컬보다 몇 배 느린 것을 확인했다.
- Vercel 응답 헤더에서 함수 실행 region을 확인했다.

원인:

- 배포 함수가 미국 region에서 실행되어 Supabase와 사용자 사이 왕복 지연이 컸다.
- 일부 API는 여러 데이터 조회가 묶여 있어 첫 응답이 길어졌다.

해결:

- `vercel.json`에서 Function region을 `hnd1`로 고정했다.
- Supabase admin client 재사용 등 서버 쪽 불필요한 비용을 줄였다.
- 하단 탭에서 필요한 API를 prefetch/cache해 반복 이동 체감을 개선했다.

결과 예시:

| API | 개선 전 평균 | 개선 후 평균 | 개선 |
| --- | ---: | ---: | ---: |
| `/api/fridge/items` | 1664ms | 187ms | -89% |
| `/api/family` | 2296ms | 561ms | -76% |
| `/api/home/summary` | 2965ms | 1025ms | -65% |

면접 답변:

> 처음에는 Lighthouse LCP를 보고 화면 렌더링 문제라고 의심했지만, 실제 클릭 지연은 API 응답 시간이 더 큰 원인이었습니다. 그래서 Lighthouse뿐 아니라 로그인 세션으로 API별 응답 시간을 따로 측정했습니다. 배포 함수 region이 멀리 잡혀 있는 것을 확인하고 `hnd1`로 고정한 뒤 주요 API 응답 시간이 크게 줄었습니다. 이 경험을 통해 성능 문제는 점수 하나가 아니라 실제 사용자 흐름과 네트워크 타이밍을 나눠 봐야 한다는 걸 배웠습니다.

## AI 품질 검증 스토리

문제:

- AI 추천은 그럴듯한 문장을 만들 수 있지만, 아이 식단에서는 안전성과 일관성이 중요하다.
- 재료, 월령, 알레르기 같은 조건을 무시하면 기능 신뢰도가 떨어진다.

해결:

- 평가 케이스 JSON을 만들고, 추천 결과를 케이스별로 점검했다.
- 스크립트로 AI 성능/품질 리포트를 생성해 이력을 남겼다.
- 보호자가 최종 확인해야 한다는 UX와 안내를 유지했다.

면접 답변:

> AI 기능은 API를 붙이는 것보다 결과를 어떻게 신뢰할 수 있게 관리하느냐가 중요했습니다. 냠픽에서는 레시피 추천을 만들 때 평가 케이스를 두고, 결과가 조건을 잘 반영하는지 리포트로 확인했습니다. 특히 아이 식단 도메인이라 알레르기나 월령 같은 조건은 보호자 확인이 필요하다는 전제를 두고 설계했습니다.

## 예상 질문과 답변

### 왜 Next.js를 썼나요?

> 로그인 기반 앱 화면과 검색 노출이 필요한 공개 페이지가 함께 있었기 때문입니다. React로 앱 UX를 만들면서도 랜딩과 가이드 페이지는 정적 HTML로 만들어 SEO에 유리하게 구성할 수 있었습니다.

### 왜 루트 `/`를 랜딩으로 바꿨나요?

> 검색엔진과 신규 사용자가 처음 만나는 대표 주소가 `/`이기 때문입니다. 기존에는 `/`가 로그인 뒤 식단 홈 역할이라 비로그인 상태에서는 앱 로딩 흐름이 먼저 보였습니다. 그래서 공개 랜딩을 `/`로 옮기고, 로그인 뒤 홈은 `/meal`로 분리했습니다.

### robots.txt에서 로그인 페이지를 막은 이유는?

> 검색 결과에 로그인 뒤 앱 화면이나 API가 노출될 필요가 없기 때문입니다. 검색엔진에는 공개 콘텐츠만 보여주고, 인증이 필요한 기능 화면은 제외했습니다.

### Lighthouse 점수가 좋은데 사용자는 왜 느리다고 느꼈나요?

> Lighthouse navigation 점수는 첫 페이지 로드 기준이고, 사용자가 말한 불편은 탭 클릭 후 API 응답 대기였습니다. 그래서 Lighthouse와 별개로 API latency를 측정했고, 배포 region 문제가 큰 원인임을 확인했습니다.

### Skeleton UI 때문에 느린 건 아니었나요?

> Skeleton은 대기 중 빈 화면을 줄이는 UI일 뿐이고, 실제 지연의 원인은 데이터가 늦게 오는 것이었습니다. Skeleton이 오래 보인다면 Skeleton 자체보다 API 응답 시간과 캐싱 전략을 봐야 합니다.

### AI 추천 품질은 어떻게 보장했나요?

> 완전한 보장보다는 품질 기준을 두고 반복 검증하는 방식으로 접근했습니다. 평가 케이스와 리포트를 만들어 조건 반영 여부를 확인했고, 도메인 특성상 보호자 최종 확인을 전제로 UX를 설계했습니다.

### 프로젝트에서 가장 어려웠던 문제는?

> 배포 환경에서만 느린 성능 문제였습니다. 로컬에서는 빠르게 보여 원인을 바로 알기 어려웠고, Lighthouse만으로도 설명이 부족했습니다. API별 응답 시간과 배포 region을 확인하면서 실제 병목을 분리했고, region 설정과 API 비용 절감으로 개선했습니다.

### 지금 다시 한다면 무엇을 개선하겠나요?

> 서버 API 내부 timing 로그를 더 일찍 넣고, `/api/home/summary`처럼 여러 데이터를 모으는 endpoint는 쿼리 수와 병렬화를 처음부터 더 엄격하게 설계했을 것 같습니다. SEO도 초기부터 `/` 공개 랜딩과 앱 홈을 분리했을 것입니다.

## 연습 순서

1. 1분 프로젝트 소개를 외운다.
2. Next.js/SSR/CSR/SSG 설명을 냠픽 라우트 예시로 말한다.
3. 성능 개선 스토리를 문제-측정-원인-해결-결과 순서로 말한다.
4. AI 추천 품질 검증을 "API 연결"이 아니라 "신뢰성 관리" 관점으로 말한다.
5. SEO 개선을 "공개 콘텐츠와 로그인 앱 분리" 관점으로 말한다.
6. 마지막으로 한계와 개선점을 솔직하게 정리한다.
