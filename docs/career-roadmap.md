# 냠픽 취업 목표 기반 개선 로드맵

## 최종 목표

냠픽을 단순 개인 프로젝트가 아니라, 서류와 면접에서 "문제를 정의하고, 측정하고, AI/웹 기술로 개선한 경험"으로 설명할 수 있는 프로젝트로 만든다.

최종적으로 이력서에는 다음 역량이 드러나야 한다.

- 사용자 문제를 제품 기능으로 정의하는 능력
- AI API를 단순 호출이 아니라 서비스 흐름으로 오케스트레이션하는 능력
- 성능 병목을 측정하고 개선하는 능력
- 인증/권한/데이터 흐름을 설계하는 능력
- SEO, 배포, CI 등 실제 서비스 운영 흐름을 이해하는 능력
- 개선 결과를 수치와 문서로 남기는 습관

## 프로젝트 포지셔닝

냠픽은 아이 식단 기록, 냉장고 재료 관리, 영수증 스캔, AI 레시피 추천을 연결한 모바일 식단 관리 서비스다.

면접과 이력서에서는 "육아 도메인 앱"보다 아래 방향으로 설명한다.

> AI를 활용해 사용자의 입력 비용을 줄이고, 냉장고 재료와 식단 기록을 기반으로 개인화된 유아식 추천 흐름을 만든 서비스

강조할 키워드:

- AI 오케스트레이션
- 영수증 OCR 기반 입력 자동화
- 냉장고 재료 기반 추천
- AI 품질 검증
- API 성능 개선
- SEO와 공개 콘텐츠 분리
- CI 기반 품질 검증

## 현재 문제 상황 정의

| 문제 | 현재 상태 | 취업 관점에서 부족한 점 |
| --- | --- | --- |
| AI 기능이 "호출했다" 수준으로 보일 수 있음 | 영수증 OCR, 레시피 추천, 품질 테스트가 있음 | AI 오케스트레이션과 품질 관리 구조를 더 명확히 보여줘야 함 |
| 추천 품질을 더 설명할 근거가 필요함 | 평가 케이스와 리포트가 있음 | 실패 케이스 관리, 회귀 방지, 지표 기준이 더 선명해야 함 |
| 사용자의 반복 사용 가치가 약하게 보일 수 있음 | 식단/냉장고/레시피 기능이 연결됨 | 개인화와 피드백 반영이 있으면 더 강함 |
| 운영 경험이 부족해 보일 수 있음 | Vercel 배포, 성능 측정, SEO 정리 완료 | CI, 배포 검증, 품질 gate를 더 명확히 자동화해야 함 |
| 이력서 성과 문장이 추상적일 수 있음 | 여러 개선을 했지만 흩어져 있음 | 문제-행동-결과 수치로 정리해야 함 |

## 방향성

### 1. AI를 잘 쓰는 개발자로 보이기

목표:

> AI API를 붙이는 개발자가 아니라, AI 결과를 검증하고 서비스 데이터로 안전하게 연결하는 개발자로 포지셔닝한다.

냠픽에서 보여줄 수 있는 구조:

```text
입력 검증
→ 프롬프트 구성
→ OpenAI 호출
→ JSON/구조화 응답 파싱
→ 품질 검증
→ fallback
→ rate limit / token budget
→ 사용자 확인
→ 서비스 데이터 저장
→ 리포트/테스트로 품질 추적
```

해야 할 개선:

- Structured Outputs 또는 Zod schema 기반 AI 응답 검증
- 재료 정규화 사전 추가
- AI 추천 결과 캐싱
- 사용자 피드백 기반 추천 가중치
- AI 품질 gate를 CI에 포함

### 2. 실제 서비스 운영을 아는 개발자로 보이기

목표:

> 만들고 끝낸 프로젝트가 아니라, 배포 후 성능과 검색 노출, 품질 자동화까지 관리한 프로젝트로 만든다.

냠픽에서 이미 한 것:

- Vercel 배포
- Function region 개선
- API latency 측정
- Lighthouse 측정
- SEO 공개 페이지 분리
- sitemap/robots 관리

해야 할 개선:

- GitHub Actions CI 정착
- PR마다 lint/test/build 자동 검증
- AI quality gate를 CI에 안전하게 연결
- 배포 후 smoke check 문서화

### 3. 제품 문제를 정의하는 개발자로 보이기

목표:

> "기능을 만들었다"가 아니라 "사용자 문제를 줄였다"로 설명한다.

냠픽의 사용자 문제:

| 사용자 문제 | 냠픽 해결 방향 |
| --- | --- |
| 매번 아이 식단을 기억하기 어렵다 | 날짜별 식단 기록 |
| 냉장고 재료를 보고 메뉴를 떠올리기 어렵다 | 재료 기반 AI 레시피 추천 |
| 장 본 재료를 수동 입력하기 번거롭다 | 영수증 스캔 OCR |
| AI 추천을 그대로 믿기 어렵다 | 금지 조합/알레르기/출처 검증 |
| 앱은 있는데 검색에서 발견되기 어렵다 | 공개 랜딩/가이드/SEO 정리 |

## 우선순위 로드맵

### 1단계: CI 품질 자동화

목표:

> push/PR마다 기본 품질을 자동 검증한다.

작업:

- GitHub Actions에서 `npm ci`, `npm run lint`, `npm run test:unit`, `npm run build` 실행
- `main`, `dev`, PR 대상 적용
- `main` branch protection으로 PR과 CI 통과를 merge 조건으로 설정
- README나 면접 문서에 CI 구조 설명 추가

이력서 문장 후보:

> GitHub Actions 기반 CI를 구성해 PR/push마다 lint, unit test, production build를 자동 검증하도록 개선

면접 포인트:

- CI는 자동 검사, CD는 Vercel 배포로 역할 분리
- AI 실호출 테스트는 비용/키 문제 때문에 unit quality gate와 분리

### 2단계: AI 응답 구조화

목표:

> AI 응답 파싱 실패와 필드 누락 가능성을 줄인다.

작업:

- 레시피 추천 응답 schema 정의
- `title`, `subtitle`, `taste`, `ingredients`, `steps`, `sourceName`, `sourceUrl` 필수/선택 기준 정리
- 현재 수동 `JSON.parse` 이후 검증 흐름을 schema 검증으로 정리
- 실패 시 fallback 호출 조건 명확화

이력서 문장 후보:

> AI 레시피 추천 응답을 schema 기반으로 검증해 JSON 파싱 실패와 잘못된 추천 결과를 방어하는 오케스트레이션 흐름 구현

면접 포인트:

- AI는 비결정적이므로 출력 형식을 강제하고 검증해야 함
- 단순 prompt보다 schema + 서버 검증이 안정적

### 3단계: 재료 정규화 레이어

목표:

> 영수증 OCR과 레시피 추천 입력 품질을 높인다.

작업:

- 재료 표준명 사전 추가
- 브랜드명/수량/수식어 제거
- 예: `친환경 애호박 1개` → `애호박`
- 중복 재료 병합
- 영수증 스캔과 레시피 추천 입력 양쪽에서 재사용

이력서 문장 후보:

> 영수증 OCR 결과와 추천 입력을 표준 재료명으로 정규화해 중복 데이터를 줄이고 AI 추천 입력 품질을 개선

면접 포인트:

- AI 호출 전에 입력 데이터를 깨끗하게 만드는 것이 품질과 비용에 직접 영향
- AI로 모든 것을 해결하지 않고 로컬 규칙/사전으로 1차 처리

### 4단계: 추천 결과 캐싱

목표:

> 같은 재료 조합에 대한 AI 비용과 대기 시간을 줄인다.

작업:

- 재료 목록을 정렬/정규화해 cache key 생성
- 동일 조합은 일정 시간 캐시된 추천 반환
- 사용자/아이 월령/알레르기 조건이 들어가면 key에 포함
- 캐시 hit/miss 지표 기록

이력서 문장 후보:

> 재료 조합 기반 AI 추천 캐싱을 도입해 반복 요청의 응답 시간과 토큰 비용을 절감

면접 포인트:

- AI 서비스는 비용과 latency가 제품 품질에 직접 연결됨
- 캐싱은 UX와 운영 비용을 동시에 개선

### 5단계: 사용자 피드백 기반 개인화

목표:

> 저장/삭제/좋아요 데이터를 다음 추천에 반영한다.

작업:

- 사용자가 저장한 AI 추천 기록 분석
- 자주 저장한 재료/조리 방식 가중치 부여
- 싫어요/삭제한 추천은 유사 조합 감점
- prompt 입력에 선호/비선호 요약 추가

이력서 문장 후보:

> 사용자의 저장/삭제 피드백을 추천 입력에 반영해 개인화된 유아식 추천 흐름으로 확장

면접 포인트:

- 처음부터 복잡한 ML이 아니라 룰 기반 preference score로 시작
- 서비스 데이터가 AI 품질 개선에 다시 들어가는 구조

### 6단계: 주간 식단표 AI 초안

목표:

> 단일 레시피 추천에서 식단 계획 생성으로 확장한다.

작업:

- 냉장고 재료, 최근 식단, 아이 정보 기반 3일/7일 식단 초안 생성
- 바로 저장하지 않고 보호자가 수정 후 확정
- 알레르기/반복 재료/영양 다양성 기준 추가
- 결과를 식단 기록에 연결

이력서 문장 후보:

> 냉장고 재료와 과거 식단 기록을 기반으로 주간 식단 초안을 생성하고 보호자 검토 후 저장하는 AI 식단 계획 기능 설계

면접 포인트:

- AI 결과를 자동 확정하지 않고 human-in-the-loop 구조로 설계
- 도메인 안전성이 중요한 서비스에서는 사용자의 최종 확인이 필요

## 이력서에 들어갈 최종 문장 후보

### AI/오케스트레이션

- OpenAI Vision을 활용해 영수증 이미지에서 식재료 후보를 추출하고, 정규화/중복 제거/사용자 확인 후 냉장고 데이터로 저장하는 AI OCR 흐름 구현
- 냉장고 재료 기반 유아식 레시피 추천에서 JSON 파싱, 금지 조합 필터링, 알레르기 주의 문구 검증, fallback 생성, token budget/rate limit을 포함한 AI 오케스트레이션 구현
- AI 추천 품질 평가 케이스와 리포트 스크립트를 작성해 추천 결과의 유효성, 출처, 알레르기 안전성, 금지 조합 회귀를 추적

### 성능/운영

- 배포 환경 API latency를 endpoint별로 측정하고 Vercel Function region을 `hnd1`로 고정해 주요 API 평균 응답 시간을 최대 89% 개선
- Lighthouse와 API latency를 분리 측정해 사용자 클릭 지연의 실제 병목이 화면 렌더링이 아닌 서버/API 응답임을 식별
- GitHub Actions CI를 구성해 PR/push마다 lint, unit test, production build를 자동 검증

### SEO/제품화

- 검색 노출을 위해 루트 `/`를 공개 랜딩으로 전환하고 로그인 뒤 앱 홈을 `/meal`로 분리
- `sitemap.ts`, `robots.ts`, canonical URL을 정리해 공개 페이지와 인증 필요 페이지의 크롤링 정책을 분리
- 아기 식단표, 이유식 큐브 보관, 냉장고 재료 유아식 가이드 페이지를 추가해 검색 유입용 공개 콘텐츠 확장

## 원칙

- 이력서에 쓸 수 없는 기능은 우선순위를 낮춘다.
- "AI를 붙였다"보다 "AI 결과를 검증하고 제품 데이터로 연결했다"를 강조한다.
- 새 기능보다 문제 정의, 측정, 개선 결과가 먼저다.
- 수치가 있으면 반드시 남긴다.
- 비용과 실패 처리를 고려한 AI 사용 경험을 만든다.
